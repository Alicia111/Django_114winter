## 1. FBV → CBV：先把 ListView / DetailView 換掉

### 為什麼要換？

| 比較 | FBV | GCBV |
|------|-----|------|
| 程式碼量 | 多 | 少（基本就 3 行） |
| 出錯機率 | 高（form 處理要自己寫） | 低（內建處理） |
| 客製化 | 直觀（看得見每行邏輯） | 要靠覆寫屬性/方法 |
| 適用場景 | 邏輯特殊、一次性的 view | 標準 CRUD |

書中後半段都會改用 GCBV，這樣 form 處理可以省超多 boilerplate。

### blog/views.py（先換 List / Detail）

```python
# blog/views.py
from django.views.generic import ListView, DetailView
from .models import Post


class BlogListView(ListView):
    model = Post
    template_name = "home.html"


class BlogDetailView(DetailView):
    model = Post
    template_name = "post_detail.html"
```

### blog/urls.py 跟著換

```python
# blog/urls.py
from django.urls import path
from .views import BlogListView, BlogDetailView

urlpatterns = [
    path("post/<int:pk>/", BlogDetailView.as_view(), name="post_detail"),
    path("", BlogListView.as_view(), name="home"),
]
```

⚠️ CBV 在 urls 要寫 **`.as_view()`**，把 class 變成 callable view function。

## 2. context_object_name：模板變數要改名

### GCBV 的「預設變數名」規則

| View | 預設模板變數 |
|------|-------------|
| `ListView` | `<model>_list` → `post_list` |
| `DetailView` | `<model>` 或 `object` → `post` 或 `object` |

W7 的 `home.html` 用的是 `posts`，現在要改成 `post_list`：

```html
<!-- templates/home.html -->
{% extends "base.html" %}

{% block content %}
{% for post in post_list %}    <!-- 從 posts 改成 post_list -->
<div class="post-entry">
  <h2><a href="{{ post.get_absolute_url }}">{{ post.title }}</a></h2>
  <p class="post-author">作者：{{ post.author.name }}</p>
  <p>{{ post.body }}</p>
</div>
{% endfor %}
{% endblock content %}
```

### 想改回 `posts` 也可以

```python
class BlogListView(ListView):
    model = Post
    template_name = "home.html"
    context_object_name = "posts"   # 自訂變數名
```

書上選擇用預設值，實務上想要怎樣的命名都可以。

### DetailView 的 post / object

`DetailView` 的 context 同時提供 `post` 跟 `object` 兩個名字（指向同一筆資料），所以兩種寫法都能用：

```html
<!-- 用 post（model 名） -->
<h2>{{ post.title }}</h2>

<!-- 用 object（通用名） -->
<h2>{{ object.title }}</h2>
```

## 3. Mixins（小科普）

書中提到一個有趣的點：`ListView` 的 `context_object_name` 來自 `MultipleObjectMixin`，`DetailView` 的來自 `SingleObjectMixin`。為什麼會這樣？

### 什麼是 Mixin？

Mixin = 多重繼承（multiple inheritance），把某些可重用的行為包成一個 class，讓其他 class 「混進去」用。

```
ListView 的繼承鏈（從下往上）
────────────────────────────────
1. ListView                        ← 我們用的
2. MultipleObjectTemplateResponseMixin
3. TemplateResponseMixin
4. BaseListView
5. MultipleObjectMixin             ← context_object_name 在這
6. ContextMixin
7. View                            ← 所有 CBV 的最終祖先
```

### 不用背

新手不需要記這些 mixin 的名字。書上的說法是：等到要客製某個行為時，再去查它在哪一層 mixin、哪個屬性可以覆寫。

延伸閱讀：[Classy Class-Based Views](https://ccbv.co.uk/) — 把所有 GCBV 的繼承鏈跟可用屬性都列出來。

## 4. CreateView：新增文章 ★

### blog/views.py

```python
# blog/views.py
from django.views.generic import ListView, DetailView
from django.views.generic.edit import CreateView   # 新

from .models import Post


class BlogListView(ListView):
    model = Post
    template_name = "home.html"


class BlogDetailView(DetailView):
    model = Post
    template_name = "post_detail.html"


class BlogCreateView(CreateView):                  # 新
    model = Post
    template_name = "post_new.html"
    fields = ["title", "author", "body"]
```

### CreateView 的三件套

| 屬性 | 說明 |
|------|------|
| `model` | 要 create 哪個 model |
| `template_name` | 用哪個模板（裡面要有 `<form>`） |
| `fields` | form 上要顯示哪些欄位（會自動產生對應的 input） |

→ form 渲染、CSRF、儲存資料、redirect 全部 Django 幫你做。

### blog/urls.py 加路由

```python
# blog/urls.py
from django.urls import path
from .views import BlogListView, BlogDetailView, BlogCreateView   # 加 BlogCreateView

urlpatterns = [
    path("post/new/", BlogCreateView.as_view(), name="post_new"),       # 新
    path("post/<int:pk>/", BlogDetailView.as_view(), name="post_detail"),
    path("", BlogListView.as_view(), name="home"),
]
```

### templates/post_new.html

```html
<!-- templates/post_new.html -->
{% extends "base.html" %}

{% block content %}
<h1>New post</h1>
<form action="" method="post">{% csrf_token %}
  {{ form }}
  <input type="submit" value="Save">
</form>
{% endblock content %}
```

### Form 模板的 5 個關鍵

| 寫法 | 用途 |
|------|------|
| `{% extends "base.html" %}` | 繼承 layout |
| `<form method="post">` | 「送資料」用 POST（搜尋等取資料才用 GET） |
| `{% csrf_token %}` | Django 防止 **CSRF**（跨站請求偽造）的必填 token |
| `{{ form }}` | 自動渲染 fields 中宣告的所有欄位 |
| `<input type="submit">` | 送出按鈕 |

⚠️ **`{% csrf_token %}` 是強制要求**，所有 Django 的 POST form 都一定要加，不然會被擋下來（403 Forbidden）。

### Submit 之後會發生什麼？

1. Django 收到 POST 資料
2. 自動驗證每個欄位（CharField 不能空、ForeignKey 必須存在等）
3. 驗證通過 → `Post.objects.create(...)` 存進 DB
4. 因為 Model 有 `get_absolute_url()`，**自動跳轉到該文章詳情頁**

→ 看不到 redirect 寫在哪？因為 W7 在 `Post` model 裡寫了 `get_absolute_url()`，CreateView 預設會用它當 success URL。**這就是 `get_absolute_url` 的價值**。

### base.html 加「+ New Blog Post」連結

```html
<!-- templates/base.html（header 區域） -->
<header>
  <div class="nav-left">
    <h1><a href="{% url 'home' %}">Django blog</a></h1>
  </div>
  <div class="nav-right">
    <a href="{% url 'post_new' %}">+ New Blog Post</a>   <!-- 新 -->
  </div>
</header>
```

## 5. UpdateView：編輯文章

### blog/views.py

```python
from django.views.generic.edit import CreateView, UpdateView   # 加 UpdateView

class BlogUpdateView(UpdateView):
    model = Post
    template_name = "post_edit.html"
    fields = ["title", "body"]   # 注意：不含 author
```

### 為什麼 `fields` 沒有 `author`？

| 動作 | 通常該不該改 author？ |
|------|----------------------|
| Create（新增） | 要選作者 → `fields` 含 `author` |
| Update（編輯） | 通常作者不變 → `fields` 不含 `author` |

→ 想固定哪些欄位可改，就把那些放 `fields`；剩下的會保留原值。

### blog/urls.py

```python
from .views import (
    BlogListView,
    BlogDetailView,
    BlogCreateView,
    BlogUpdateView,   # 新
)

urlpatterns = [
    path("post/new/", BlogCreateView.as_view(), name="post_new"),
    path("post/<int:pk>/", BlogDetailView.as_view(), name="post_detail"),
    path("post/<int:pk>/edit/", BlogUpdateView.as_view(), name="post_edit"),  # 新
    path("", BlogListView.as_view(), name="home"),
]
```

→ 編輯頁需要 **知道改哪一筆**，所以路由帶 `<int:pk>`。

### templates/post_edit.html

```html
<!-- templates/post_edit.html -->
{% extends "base.html" %}

{% block content %}
<h1>Edit post</h1>
<form action="" method="post">{% csrf_token %}
  {{ form }}
  <input type="submit" value="Update">
</form>
{% endblock content %}
```

→ 跟 `post_new.html` 幾乎一樣，只差按鈕文字（"Save" → "Update"）跟 h1。

### post_detail.html 加 Edit 連結

```html
<!-- templates/post_detail.html -->
{% extends "base.html" %}

{% block content %}
<div class="post-entry">
  <h2>{{ post.title }}</h2>
  <p class="post-author">作者：{{ post.author.name }}</p>
  <p>{{ post.body }}</p>
</div>
<div>
  <p><a href="{% url 'post_edit' post.pk %}">+ Edit Blog Post</a></p>
</div>
{% endblock content %}
```

→ `{% url 'post_edit' post.pk %}` 會生成 `/post/1/edit/`（pk 由 template 傳入）。

### UpdateView 的「魔法」

打開編輯頁時，**form 會自動帶入該文章的現有資料**。
這件事 Django 幫你做了：

1. 讀取 URL 上的 `pk`
2. 用 `Post.objects.get(pk=pk)` 取得物件
3. 把資料填回 form 的每個欄位

→ 完全不用寫 `instance=...` 或讀資料庫的程式碼。

## 6. DeleteView：刪除文章 ★

### blog/views.py

```python
from django.views.generic.edit import CreateView, UpdateView, DeleteView   # 加 DeleteView
from django.urls import reverse_lazy   # 新


class BlogDeleteView(DeleteView):
    model = Post
    template_name = "post_delete.html"
    success_url = reverse_lazy("home")
```

### `reverse` vs `reverse_lazy` ⭐

| 函式 | 何時被執行 |
|------|----------|
| `reverse("home")` | **馬上**執行，把 name 換成路徑（e.g. `"/"`） |
| `reverse_lazy("home")` | **延遲**執行，等真的需要時才換 |

#### 為什麼這裡要用 `reverse_lazy`？

`success_url` 是**類別屬性**（class attribute），class 一被載入，這行就會執行。
但這時候 Django 的 URL dispatcher **可能還沒準備好**（還沒讀完所有 urls.py），用 `reverse` 會炸掉。

→ `reverse_lazy` 把「換算 URL」這件事**延後到真的要用的時候**才做，安全。

#### 簡單記憶

```
class 屬性裡寫 URL  →  reverse_lazy   ← 因為類別載入時 URL system 可能還沒好
function 裡寫 URL  →  reverse        ← 函式被呼叫時，所有東西都準備好了
```

### CreateView / UpdateView 為什麼不用 `success_url`？

```
CreateView / UpdateView：
  ↳ 完成後查 model 有沒有 get_absolute_url()
  ↳ 有 → 自動跳到該物件的詳情頁
  ↳ 沒有 → 報錯，要你寫 success_url

DeleteView：
  ↳ 物件都被刪了！沒有「該物件的詳情頁」可以跳
  ↳ 必須明確指定 success_url
```

→ 這就是為什麼 Delete 一定要寫 `success_url`，但 Create / Update 可以省略。

### blog/urls.py

```python
from .views import (
```

### templates/post_delete.html

### post_detail.html 加 Delete 連結

## 7. 完整的 views.py（5 個 CBV 一起看）

```python
# blog/views.py
from django.views.generic import ListView, DetailView
from django.views.generic.edit import CreateView, UpdateView, DeleteView
from django.urls import reverse_lazy

from .models import Post


class BlogListView(ListView):
    model = Post
    template_name = "home.html"


class BlogDetailView(DetailView):
    model = Post
    template_name = "post_detail.html"


class BlogCreateView(CreateView):
    model = Post
    template_name = "post_new.html"
    fields = ["title", "author", "body"]


class BlogUpdateView(UpdateView):
    model = Post
    template_name = "post_edit.html"
    fields = ["title", "body"]


class BlogDeleteView(DeleteView):
    model = Post
    template_name = "post_delete.html"
    success_url = reverse_lazy("home")
```

→ 5 個 view，加起來不到 30 行，全 CRUD 完成。

## 8. Tests（新增 3 個測試）

W7 已經有 5 個測試，這次再加 3 個對應 Create / Update / Delete：

```python
# blog/tests.py（在原本的 BlogTests 類別裡加）

def test_post_createview(self):
    response = self.client.post(
        reverse("post_new"),
        {
            "title": "New title",
            "body": "New text",
            "author": self.author.id,
        },
    )
    self.assertEqual(response.status_code, 302)   # 302 = redirect
    self.assertEqual(Post.objects.last().title, "New title")
    self.assertEqual(Post.objects.last().body, "New text")

def test_post_updateview(self):
    response = self.client.post(
        reverse("post_edit", args="1"),
        {
            "title": "Updated title",
            "body": "Updated text",
        },
    )
    self.assertEqual(response.status_code, 302)
    self.assertEqual(Post.objects.last().title, "Updated title")
    self.assertEqual(Post.objects.last().body, "Updated text")

def test_post_deleteview(self):
    response = self.client.post(reverse("post_delete", args="1"))
    self.assertEqual(response.status_code, 302)
```

### 測試重點

| 寫法 | 說明 |
|------|------|
| `self.client.post(...)` | 模擬使用者送 POST 表單 |
| 第二個參數（dict） | 表單資料（key 對應 fields） |
| `status_code == 302` | 表單成功會 redirect，**不是** 200 |
| `Post.objects.last()` | 拿剛剛新增/更新的最後一筆 |
| `args="1"` | 帶入 URL 參數 `pk=1` |

## 重點整理

```
GCBV editing views（django.views.generic.edit）
─────────────────────────────────────────────
CreateView   → 新增。需要 model + template_name + fields
UpdateView   → 編輯。同上，會自動帶入現有資料
DeleteView   → 刪除。多需要 success_url（用 reverse_lazy）

URL 反查兩種寫法
─────────────────────────────────────────────
reverse("home")        → function 內、執行時才呼叫 → 用這個
reverse_lazy("home")   → class 屬性、要延後執行   → 用這個

跳轉去哪？
─────────────────────────────────────────────
Create / Update 成功 → model.get_absolute_url() 自動跳
Delete 成功         → 必須自己寫 success_url

模板 form 的 5 件套
─────────────────────────────────────────────
{% extends "base.html" %}
<form action="" method="post">
  {% csrf_token %}      ← 必填
  {{ form }}            ← 自動渲染 fields
  <input type="submit" value="...">
</form>

GCBV 模板變數預設名
─────────────────────────────────────────────
ListView    → <model>_list   （Post → post_list）
DetailView  → <model> 或 object（Post → post 或 object）
要改名 → context_object_name = "..."

測試 form
─────────────────────────────────────────────
self.client.post(reverse("..."), {欄位 dict})
status_code 預期是 302（redirect），不是 200
```

## CRUD 對照表

| 動作 | URL | View | Template | 名稱（name=） |
|------|-----|------|----------|--------------|
| **C**reate（新增） | `/post/new/` | `BlogCreateView` | `post_new.html` | `post_new` |
| **R**ead - 列表 | `/` | `BlogListView` | `home.html` | `home` |
| **R**ead - 詳情 | `/post/1/` | `BlogDetailView` | `post_detail.html` | `post_detail` |
| **U**pdate（編輯） | `/post/1/edit/` | `BlogUpdateView` | `post_edit.html` | `post_edit` |
| **D**elete（刪除） | `/post/1/delete/` | `BlogDeleteView` | `post_delete.html` | `post_delete` |

→ 這是 Django blog 最常見的 5 個頁面，往後幾乎所有專案都會看到類似結構。
