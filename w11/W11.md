## 3. User Profile vs Custom User Model

兩種擴充使用者的方式，差別要分清楚：

| 方法 | 做法 | 特色 |
|------|------|------|
| **User Profile** | 用 `OneToOneField` 接到另一個 `Profile` model | 認證邏輯留在 `User`，擴充欄位放外面，較鬆耦合 |
| **Custom User Model**（本書採用） | 繼承 `AbstractUser`，直接在同一個 model 加欄位 | 結構單純，官方推薦新專案這樣做 |

> 兩個其實也能混用（自訂 User + 額外 Profile model），但這章先單純用 `AbstractUser`。

### `AbstractUser` vs `AbstractBaseUser`

- **`AbstractUser`**（推薦給初學者）→ 已經包好 username、email、password、permission 等，
  只要繼承後加欄位即可。
- **`AbstractBaseUser`** → 從零開始，全部要自己刻，連密碼欄位都要。
  彈性最大，但很容易翻車。除非你超清楚自己在做什麼，否則不要碰。

## 4. AbstractUser — 四個步驟

> ⭐ 整章的核心 SOP，背起來

1. 修改 `django_project/settings.py`
2. 新增 `CustomUser` model
3. 新增 `UserCreationForm` / `UserChangeForm`
4. 修改 `accounts/admin.py`

### 步驟 1：`settings.py`

```python
# django_project/settings.py
INSTALLED_APPS = [
    "django.contrib.admin",
    "django.contrib.auth",
    "django.contrib.contenttypes",
    "django.contrib.sessions",
    "django.contrib.messages",
    "django.contrib.staticfiles",
    "accounts",  # new
]

# ↓ 放在 settings.py 檔案的尾巴
AUTH_USER_MODEL = "accounts.CustomUser"  # new
```

> `AUTH_USER_MODEL` 的格式是 `"<app_label>.<ModelName>"`，**沒有引號的話會出錯**。

### 步驟 2：`accounts/models.py`

```python
from django.contrib.auth.models import AbstractUser
from django.db import models


class CustomUser(AbstractUser):
    age = models.PositiveIntegerField(null=True, blank=True)
```

#### `null=True` vs `blank=True` ← 超常考！

| 參數 | 作用層級 | 意思 |
|------|----------|------|
| `null=True` | **資料庫層** | 該欄位允許存 `NULL` |
| `blank=True` | **表單驗證層** | 該欄位允許表單留空（沒填也能送出） |

> 兩個經常**一起出現**，這樣 form 可以留空，DB 就存 NULL。

⚠️ **字串欄位（CharField / TextField）的雷**：
Django 官方慣例是「字串就用空字串 `""`」表示「沒資料」，**不要同時設 `null=True`**，
否則 DB 會出現兩種「沒資料」(`NULL` 跟 `""`)，超亂。

### 步驟 3：`accounts/forms.py`（新檔案）

```python
from django.contrib.auth.forms import UserCreationForm, UserChangeForm

from .models import CustomUser


class CustomUserCreationForm(UserCreationForm):
    class Meta:
        model = CustomUser
        fields = UserCreationForm.Meta.fields + ("age",)


class CustomUserChangeForm(UserChangeForm):
    class Meta:
        model = CustomUser
        fields = UserChangeForm.Meta.fields
```

- `UserCreationForm`：給「新註冊」用的（會處理 password1/password2 兩次輸入）
- `UserChangeForm`：給「修改既有使用者」用（admin 改使用者資料時用）
- `Meta.fields + ("age",)`：把預設欄位再加上我們的 `age`

> ❓ 為什麼 `fields` 預設只有 username/email/password？
> 因為 `UserCreationForm.Meta.fields` 預設就是這三個，雖然 `User` model 本身還有更多欄位。
> Form 的「欄位」跟 Model 的「欄位」**不是同一件事**，這個觀念剛開始很容易卡住。

### 步驟 4：`accounts/admin.py`

```python
from django.contrib import admin
from django.contrib.auth.admin import UserAdmin

from .forms import CustomUserCreationForm, CustomUserChangeForm
from .models import CustomUser


class CustomUserAdmin(UserAdmin):
    add_form = CustomUserCreationForm    # 新增使用者時用的 form
    form = CustomUserChangeForm          # 修改使用者時用的 form
    model = CustomUser
    list_display = [
        "email",
        "username",
        "age",
        "is_staff",
    ]
    # 編輯既有使用者時，多顯示 age 欄位
    fieldsets = UserAdmin.fieldsets + ((None, {"fields": ("age",)}),)
    # 新增使用者時，多顯示 age 欄位
    add_fieldsets = UserAdmin.add_fieldsets + ((None, {"fields": ("age",)}),)


admin.site.register(CustomUser, CustomUserAdmin)
```

| 屬性 | 用途 |
|------|------|
| `list_display` | Users 列表頁顯示哪些欄位 |
| `fieldsets` | 「編輯既有使用者」頁面上要不要列出該欄位 |
| `add_fieldsets` | 「新增使用者」頁面上要不要列出該欄位 |

> `fieldsets` 跟 `add_fieldsets` 都要加，不然 admin 介面看不到 `age`。

## 6. Tests — 確認 user 跟 superuser 都能正確建立

`accounts/tests.py`：

```python
from django.contrib.auth import get_user_model
from django.test import TestCase


class UsersManagersTests(TestCase):
    def test_create_user(self):
        User = get_user_model()
        user = User.objects.create_user(
            username="testuser",
            email="testuser@example.com",
            password="testpass1234",
        )
        self.assertEqual(user.username, "testuser")
        self.assertEqual(user.email, "testuser@example.com")
        self.assertTrue(user.is_active)
        self.assertFalse(user.is_staff)
        self.assertFalse(user.is_superuser)

    def test_create_superuser(self):
        User = get_user_model()
        admin_user = User.objects.create_superuser(
            username="testsuperuser",
            email="testsuperuser@example.com",
            password="testpass1234",
        )
        self.assertEqual(admin_user.username, "testsuperuser")
        self.assertEqual(admin_user.email, "testsuperuser@example.com")
        self.assertTrue(admin_user.is_active)
        self.assertTrue(admin_user.is_staff)
        self.assertTrue(admin_user.is_superuser)
```

### 重要 API

| 寫法 | 用途 |
|------|------|
| `get_user_model()` | **永遠**用這個拿 User，不要直接 import `User` |
| `User.objects.create_user(...)` | 建立一般使用者 |
| `User.objects.create_superuser(...)` | 建立超級使用者 |

| 欄位 | 一般 User | Superuser |
|------|-----------|-----------|
| `is_active` | ✅ True | ✅ True |
| `is_staff` | ❌ False | ✅ True |
| `is_superuser` | ❌ False | ✅ True |

### Step 1. 建 app

### Step 2. `settings.py`

```python
INSTALLED_APPS = [
    ...

TEMPLATES = [
    {
        ...
        'DIRS': [BASE_DIR / "templates"],   # 改這行
        ...

```

### Step 3. `articles/models.py`

```python
from django.conf import settings
from django.db import models
from django.urls import reverse


class Article(models.Model):
    title = models.CharField(max_length=200)
    body = models.TextField()
    date = models.DateTimeField(auto_now_add=True)
    author = models.ForeignKey(
        settings.AUTH_USER_MODEL,   # ⭐ 不要直接 import CustomUser
        on_delete=models.CASCADE,
    )

    def __str__(self):
        return self.title

    def get_absolute_url(self):
        return reverse("article_detail", kwargs={"pk": self.pk})
```

> ⭐ **重要觀念**：要參照「目前的 User Model」時，**永遠用 `settings.AUTH_USER_MODEL`**，
> 不要直接 `from accounts.models import CustomUser`。
> 原因跟 `get_user_model()` 一樣：保持彈性，未來換 User Model 也不用改 ForeignKey。

| 場景 | 用什麼 |
|------|--------|
| 在 **model 檔案**裡定義 ForeignKey | `settings.AUTH_USER_MODEL`（字串） |
| 在 **view / form / 其他 Python 程式碼**拿 User class | `get_user_model()` |
| **絕對不要** | `from accounts.models import CustomUser` 直接 import |

### Step 4. `articles/admin.py`

```python
from django.contrib import admin
from .models import Article


class ArticleAdmin(admin.ModelAdmin):

admin.site.register(Article, ArticleAdmin)
```

### Step 5. `articles/views.py`（CBV 五件組）

```python
from django.urls import reverse_lazy
from django.views.generic import ListView, DetailView
from django.views.generic.edit import CreateView, UpdateView, DeleteView

from .models import Article


class ArticleListView(ListView):
    model = Article
    template_name = "article_list.html"


class ArticleDetailView(DetailView):
    model = Article
    template_name = "article_detail.html"


class ArticleCreateView(CreateView):
    model = Article
    template_name = "article_new.html"
    fields = ["title", "body", "author"]


class ArticleUpdateView(UpdateView):
    model = Article
    template_name = "article_edit.html"
    fields = ["title", "body"]   # 不讓改 author


class ArticleDeleteView(DeleteView):
    model = Article
    template_name = "article_delete.html"
    success_url = reverse_lazy("article_list")
```

### Step 6. `articles/urls.py`

```python
from django.urls import path
from .views import (
    ArticleListView, ArticleDetailView,
    ArticleCreateView, ArticleUpdateView, ArticleDeleteView,
)

urlpatterns = [
    path("", ArticleListView.as_view(), name="article_list"),
    path("new/", ArticleCreateView.as_view(), name="article_new"),
    path("<int:pk>/", ArticleDetailView.as_view(), name="article_detail"),
    path("<int:pk>/edit/", ArticleUpdateView.as_view(), name="article_edit"),
    path("<int:pk>/delete/", ArticleDeleteView.as_view(), name="article_delete"),
]
```

### Step 7. `django_project/urls.py`

```python
from django.contrib import admin
from django.urls import path, include
from django.views.generic import RedirectView

urlpatterns = [
    path('admin/', admin.site.urls),
    path('articles/', include('articles.urls')),
    path('', RedirectView.as_view(url='/articles/', permanent=False)),  # 首頁→文章列表
]
```

> 💡 為什麼最後一行要加 `RedirectView`？
> 因為我們把文章 CRUD 全部放在 `/articles/` 底下，根目錄 `/` 沒接任何 view，
> 直接打 `http://127.0.0.1:8000/` 會 404。
> 用 `RedirectView` 把 `/` 自動轉到 `/articles/`，就是首頁的感覺了。
>
> `permanent=False` 代表是 **302 暫時重導**，不是 301 永久重導 —
> 開發階段建議用 302，避免瀏覽器把 URL 永久 cache 起來，後面想改首頁就麻煩了。

## 12. CSS / Static Files 美化

書本到 Ch11 才會教 Bootstrap，但我們先用手寫 CSS 練一下 Django 的 **static files** 機制。

### Step 1. `settings.py`

```python
STATIC_URL = 'static/'
STATICFILES_DIRS = [BASE_DIR / "static"]   # 新增這行
```

| 設定 | 用途 |
|------|------|
| `STATIC_URL` | 瀏覽器看到的 URL 前綴（`/static/css/base.css`） |
| `STATICFILES_DIRS` | 開發時 Django **去哪裡找** static 檔（**專案層級**的 static 資料夾） |
| `STATIC_ROOT` | 上線部署時 `collectstatic` 把全部檔集中到這裡（暫時用不到） |

### Step 2. 建立資料夾

```bash
mkdir -p static/css
```

完整結構：

```
w11/
├── static/
│   └── css/
│       └── base.css     ← 手寫的 CSS
├── templates/
│   └── base.html        ← 用 {% static %} 載入
```

### Step 3. `base.html` 載入 CSS

```html
{% load static %}        ← 一定要在最上面
<html>
<head>
  <link href="{% static 'css/base.css' %}" rel="stylesheet">
  <!-- 順手掛個 Google Fonts，繁中襯線字超漂亮 -->
  <link href="https://fonts.googleapis.com/css2?family=Noto+Serif+TC:wght@400;700;900&display=swap"
        rel="stylesheet">
</head>
...
```

⭐ **`{% load static %}` 必須放在用到 `{% static %}` 的 template 最上面**，不然會噴錯。

### Step 4. 設計取向

這次的 CSS 走「**現代報紙風**」：

- **配色**：米白底 (`#fdfaf4`) + 墨黑字 + 暗紅 accent (`#8b2a1f`)，像實體報紙
- **字體**：Noto Serif TC（繁中襯線字），Header 用 uppercase 大寫拉開字距
- **卡片**：每篇文章一張小卡，左側 3px 紅線 + 淺陰影
- **表單**：label 走全大寫小字級（報紙標題感），輸入框 focus 時邊框變紅
- **CSS 變數**：`:root { --bg, --ink, --accent ... }` 集中管理顏色，要改主題色超快

### Step 5. 重新整理瀏覽器

Django 開發模式會**自動**透過 `django.contrib.staticfiles` 處理 `/static/...` 的請求，
不需要手動跑 `collectstatic`，存檔後 F5 就生效。

### 常見小坑 `(^_^;)`

| 症狀 | 原因 |
|------|------|
| CSS 沒套上去，F12 看是 404 | 忘記 `{% load static %}`，或路徑拼錯 |
| `'static' is not a registered tag library` | `{% load static %}` 沒寫在 template 開頭 |
| 改了 CSS 沒變 | 瀏覽器 cache，Ctrl + Shift + R 強制重整 |
| `STATICFILES_DIRS` 沒設，CSS 卻能載 | App 內建的 `static/` 資料夾也會被掃到，但**專案層級**的要靠這個設定 |
